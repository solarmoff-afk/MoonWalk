# Вклад в MoonWalk
Чтобы внести вклад в разработку MoonWalk вам необходимо создать форк этого репозитория, сделать коммит своих изменений (git commit -m "...") и отправить их (git push), а потом создать запрос на слияние (pull request) на GitHub с вашими изменениями (ВАЖНОЕ УСЛОВИЕ: Вы должны соблюдать все правила запросов на слияние которые прописаны ниже)

## Правила оформления коммитов
Каждый коммит в репозитории должен содержать префикс, формат коммита:

```
префикс(где именно): какие-то изменения
```

### Префиксы коммитов
- **feat**: Новая функциональность. При feat коммите необходимо сменить semver в **moonwalk/Cargo.toml** (Либо Cargo.toml другого крейта в который были внесены изменения). Изменяется именно **минорная** версия, то есть 0.X.0 где x это минорная версия. Пример: 0.2.0 -> 0.3.0
- **fix**: Это исправление бага (patch). При таком коммите необходимо изменить патч версию в semver Cargo.toml. То есть 0.0.X где X это патч версия. Пример: 0.2.0 -> 0.2.1
- **docs**: Изменения в документации. Сюда относятся изменения изменения в документации функций/структур/перечислений в rust коде (то есть `///`), изменения в README.md, STYLE_GUIDE.md, CONTRIBUTING.md либо любой другой текст, который нужен только для прочтения его человеком
- **style**: Форматирование кода. Добавление отступов, переносов строк, пробелов и так далее, всё что связано с стилисткой кода
- **refactor**: Рефакторинг кода без изменения функциональности. Распутывание спагетти-кода, разбиение на более мелкие функции и в целом упрощение кода для улучшения качества кода для улучшения его понимания
- **perf**: Улучшения производительности (Время выполненения, запуска и потребление ОЗУ/Видеопамяти)
- **test**: Добавление или изменение тестов
- **build**: Изменения в системе сборки (Добавление скриптов в папку scripts, изменения в Cargo.toml не связанные с изменением версии крейта)
- **ci**: Настройки CI (Github actions в данном случае, но также это может и означать изменения в gitlab ci и других если они будут)
- **chore**: Рутинные задачи (Обновление версий зависимостей)

### Примеры мест, где может быть изменение (не все)
batching, render, shader, android, easygpu, textware, resources, errors, objects

## Стилистика кода
Рекомендуется писать код в стилистике, которая описана в файле [STYLE_GUIDE.md](STYLE_GUIDE.md) для того, чтобы поддерживать единобразие кодовой базы. Также в начале кадого нового файла должен быть такой заголовок (О и чём написано в стайл гайде) и перенос строки после него

```rust
// Часть проекта MoonWalk с открытым исходным кодом.
// Лицензия EPL 2.0, подробнее в файле LICENSE. Copyright (c) 2025 MoonWalk
```

## Публичный API
Если вы вводите новый api, то пожалуйста, соблюдайте эти правила для разработки функций включая наименование и единобразие API

- **new_**: Добавление нового объекта, всегда возвращает только ObjectId
- **set_**: Изменяет какой-то параметр объекта. НИКОГДА ничего не возвращает и принимает Id первым аргументом
- **load_**: Загрузка какого-то файла на жёстком диске через [resource_manager.rs] (resource_manager.rs) и возврат его айди 
- **load_x_from_bytes**: Загрузка чего-то из набора байтов и возврат его айди
- **get_**: Получение какого-то параметра объекта
- **resolve_**: Передача входный данных и получение выходных. Например: Получить координаты и вернуть айди объекта который находится на них
- **Другое имя**: Перевод объекта в какое-то состояние либо просто какое-то действие
